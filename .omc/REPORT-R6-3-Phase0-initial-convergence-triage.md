## 0. 使用者校正後的 C3 / C4 定義

```
C3 = 只開 Cloud 漫射燈
C4 = 只開軌道燈 + 廣角燈，不開 Cloud 漫射燈
```

舊文件曾把 C4 寫成「C3 + 軌道 / Cloud + 軌道全開 / 10 光源」，已由 Claude 修正或加註過期警示。

重要推論：

```
C3 與 C4 不應再混成同一種多光源問題。
C3 偏 Cloud 漫射燈取樣 / 初期高變異問題。
C4 偏軌道 + 廣角燈造成的暗部光斑、反射、firefly 問題。
```

## 1. 已知使用者事實

使用者描述：

```
100 samples per pixel 內：整張都髒，螢火蟲很多。
約 100 samples per pixel 後：快速收斂到較穩定狀態。
後續主要殘留：
  1. 暗部螢火蟲 / 光斑
  2. 物體交界邊界殘留

快速預覽模式：4 bounce，約 3000 samples per pixel 可接受。
趨近真實模式：14 bounce，約 20000 samples per pixel 才全部消失。

4 bounce 跑到 20000 samples per pixel 約 30 分鐘。
14 bounce 跑到 20000 samples per pixel 約 4.5 小時。
```

由此可得：

```
14 bounce 比 4 bounce 慢約 9 倍。
這不是單純 14 / 4 = 3.5 倍的線性變慢，而是超線性變慢。
```

補充使用者截圖結論：

```
C3 14-bounce 1~3000 spp 截圖顯示，可先把問題切成兩段：

1 ~ 48 spp：
  初始爆噪問題。
  較適合用時序累積、重投影、SVGF、preview-only resolve 改善剛切換時的視覺衝擊。

48 ~ 20000 spp：
  尾端慢慢清髒點問題。
  較適合處理螢火蟲、高亮光斑、radiance / throughput / light sampling 權重。
```

## 2. 第一段量測：C1-C4 不是明顯 wall-clock 變慢

測試條件：

```
Browser: Brave 測試視窗
Viewport: 1200 x 1290
Device pixel ratio: 2
URL: http://localhost:9001/Home_Studio.html
讀值方式：DevTools Protocol 讀頁面 `sampleCounter` 與 UI 耗時
```

結果：

| Config | 256 spp | 512 spp | 1024 spp | 1024 spp 約略速度 |
|---|---:|---:|---:|---:|
| C1 | 7.193s | 14.310s | 28.145s | 36.38 spp/s |
| C2 | 7.379s | 14.288s | 28.077s | 36.47 spp/s |
| C3 | 7.576s | 15.021s | 29.426s | 34.80 spp/s |
| C4 | 7.371s | 14.864s | 29.711s | 34.47 spp/s |

結論：

```
C3 / C4 到 1024 spp 只比 C1 / C2 慢約 5%。
主要痛點不是「每秒 samples per pixel 大幅下降」。
主要痛點是「同樣 samples per pixel 下，C3 / C4 畫面變異更高、看起來更髒」。
```

## 5. 初步歸因：其實是三個問題

補充：

```
C3 14-bounce 1~3000 spp 截圖把早期問題切得更細：
1~48 spp 偏初始爆噪；48 spp 後偏尾端 firefly / 光斑拖尾。
因此下方 0~100 spp 與 100~1000 spp 是原始粗分段，不代表硬邊界。
```

### A. 0 ~ 100 spp：全圖初期爆噪

症狀：

```
整張圖到處都是螢火蟲 / 隨機亮點。
C3 特別明顯。
```

較可能方向：

```
1. 時序累積 / 重投影
   把上一 frame 可用資訊接過來，降低切換後從 0 開始的視覺衝擊。

2. SVGF（Spatiotemporal Variance-Guided Filtering，時空變異引導濾波）
   用時間 + 空間 + 變異資訊做智慧降噪。

3. 快速預覽模式專用 denoise resolve
   不必追求最終物理精準，先追求 30~60 秒內可看。
```

### B. 100 ~ 1000 spp：暗部螢火蟲 / 光斑拖尾

症狀：

```
全圖雜訊下降後，剩下暗部高亮離群點。
C4 也有，但 C3 更嚴重。
```

較可能方向：

```
1. firefly clamp（高亮離群值限制）
2. 穩健累積策略
3. 間接光路徑權重檢查
4. 光源取樣權重檢查
```

注意：

```
這類問題跟「跑得慢」不同。
它比較像少數樣本權重太大，拖慢視覺收斂。
```

### C. 高 spp 仍有少量邊界殘留

症狀：

```
物體交界 / 邊界放大看仍有少量不乾淨。
跑到 20000 spp 仍可能有細節殘留。
```

較可能方向：

```
1. 邊界抗鋸齒
2. edge-aware resolve
3. 法線 / 幾何交界穩定性檢查
```

注意：

```
這不應與一般 path tracing 雜訊混在一起處理。
```

## 6. 對 R6-3 / R7 的含義

短期 R6-3 可優先處理：

```
1. 快速預覽模式 4 bounce 的 0~100 spp 全圖爆噪
2. 100~1000 spp 暗部 firefly / 光斑拖尾
```

中期 R6-3 可評估：

```
1. SVGF（時空變異引導濾波）
2. 重投影
3. firefly clamp
4. preview-only resolve
```

可能跨入 R7 的部分：

```
14 bounce 到 20000 spp 需要 4.5 小時。
若目標是讓趨近真實模式也大幅加速，單純後處理可能不夠。
這可能需要採樣演算法升級，例如 ReSTIR / path guiding / 光源取樣權重重構。
```

## 7. 建議下一步

建議不要再大量補圖，先進入針對性 code review：

```
第一優先：讀目前累積 / resolve / screen output pipeline
  目標：判斷能否做 preview-only temporal denoise / SVGF / reproject。
  C3 14-bounce 1~48 spp 截圖補充後，此項仍是第一優先。

第二優先：讀 radiance accumulation 與 firefly 高亮路徑
  目標：判斷是否能安全加 firefly clamp，且不破壞 1024+ spp 主觀色彩。
  主要對應 48 spp 後的暗部小亮點 / 高亮光斑拖尾。

第三優先：讀邊界處理與 screen resolve
  目標：把邊界殘留跟一般雜訊分開，不要用同一把刀亂砍。
```

守門：

```
任何 shader 實作前，先產出小 plan。
不要把 R6-2 Russian Roulette 當解法；它會增加 variance，方向不對。
不要把 C3 / C4 混為同一種燈光型態。
```
