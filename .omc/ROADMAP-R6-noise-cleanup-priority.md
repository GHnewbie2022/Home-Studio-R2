## 為何要這份文件

R6-2 ROADMAP 原版聚焦「速度優化」、但使用者實際痛點是「雜訊清理」：

```
症狀 1   1000 spp 還收不乾淨
症狀 2   陰影有雜點
症狀 3   牆角螢火蟲（單像素亮到爆）
症狀 4   整體看起來髒
```

使用者明確表態「速度慢沒關係、放著等就好」、優先解決雜訊。本檔依 Karpathy 思考邏輯（K 神：先診斷、簡單先、外科手術改動、可驗證目標）排序候選方案。

```
R6-1     5×5 cross-bilateral blur                ❌ 已撤回
         原因：對均勻雜點只變糊、對螢火蟲反而保留

        正確路線必須在 in-renderer 改採樣品質、或上專業 denoiser
```


## 各方案逐項說明

### 🏆 G. NEE PDF 診斷 + 修正

```
為何最好     K 神原則 1「Think before coding」最直接體現
             成本：1~2 天診斷、若命中 30 分鐘修公式
             若 NEE PDF 對小光源不利、直接修根因、其他方案不需做

做法
  a. shader 加診斷探針（類似 R6-2 桶 2 #2 Step 0 那種）
  b. 印出每盞燈的 NEE PDF 分布
  c. 找出「PDF 過小但能量大」的取樣（螢火蟲源頭）
  d. 改 PDF 公式對症處理

風險
  診斷不出明確根因 → 排除一條路、cost 低
  命中 → 直接終結痛點
```

### 🏆 H. 場景幾何修正

```
為何好       某些螢火蟲來自「光源邊界精度」或「BVH 邊界 self-intersection」
             成本：用 debug 視覺化找出螢火蟲座標、修一兩個數字

做法
  a. 寫 shader「螢火蟲座標標記模式」、把超過某能量上限的像素位置印出來
  b. 對照場景幾何看是否集中在某個邊角
  c. 平滑該邊角 / 微調光源幾何 / 排除自交

風險
  並非所有螢火蟲都來自幾何（有的來自統計爆炸）
  若幾何沒問題、cost 半天、無傷
```

### 🥈 F. Light Tree / Light BVH

```
為何次好     對症（多光源 + 小光源 = 22W COB + 25W 軌道燈場景特性）
             比 ReSTIR 簡單、效果略遜

做法         建光源樹狀結構、高亮度 + 對該像素重要的光優先取樣
工時         1~2 週
風險         中（新資料結構、shader 整合）
何時上       G/H 診斷後確定「就是光源取樣分配不均」、F 是中等投資解
```

### 🥈 D. 現代 denoiser（OIDN / OptiX）

```
為何在這層   對症（直接壓螢火蟲 + 雜點）
             整合中等難度（JS / WASM 橋接、Three.js 取 RT buffer）
             違反 K 神「Surgical Changes」（引入外部依賴）
             但比 R6-1 bilateral blur 強很多級（AI 訓練、知道保邊砍螢火蟲）

工時         1~2 週（含整合）
風險         中（外部庫依賴、可能跟 erichlof 框架衝突）
何時上       in-renderer 改良（G/H/F/B）窮盡仍不滿足、接受後處理路線作保險
前置驗證     OIDN 在 Three.js + WebGL 有現成整合方案 → 假設 3 需先查證
```

### 🥈 B. ReSTIR

```
為何最對症   業界公認直接光雜訊最強解
             小光源 + 高能量低機率取樣會被「水塘抽樣 + 時間複用」直接砍

工時         2~3 週、複雜度高
風險         高（WebGL2 上可實作但邊角多、多數 ReSTIR 實作在 native）
何時上       「窮盡其他方案後最後手段」
             G/H 沒命中 + F 投入後仍不滿足 → 才考慮 B
前置驗證     WebGL2 ReSTIR 實作案例 → 假設 4 需先查證
K 神視角     違反原則 2「Simplicity First」、應為最後選項
```

### 🪙 A. 2-1 自適應採樣

```
對痛點       陰影雜點中等、螢火蟲有限
做法         variance estimator + ray budget 動態分配
工時         3~5 天
定位         「補強型」、做了不會錯、不是根因解
             若 G/H 已修了根因、A 收益邊際更小
             若 G/H 沒命中、A 也很難救螢火蟲（單像素極值問題）
ROI          比 F (Light Tree) 略低
```

## K 神式建議路線（對應 Goal-Driven Execution）

```
Step 1   G + H 並行診斷
         目標：找出根因 / 排除根因
         成功 = 螢火蟲源頭定位 + NEE PDF 分布出爐
         工時：2~3 天
         
Step 2-A 若 G/H 命中根因 → 修公式 / 修幾何
         成功 = 1000 spp 測試影格達標
         工時：30 分鐘 ~ 1 天
         
Step 2-B 若 G/H 沒命中 → 上 F (Light Tree)
         成功 = 螢火蟲與雜點降低到肉眼可接受
         工時：1~2 週
         前置：先設計 light tree 結構、再實作
         
Step 3   若 F 仍不滿足 → 上 B (ReSTIR) 或 D (denoiser) 二擇一
         成功 = 同 Step 2 標準
         工時：2~3 週（B）或 1~2 週（D）
         二擇一決策依據：
           - 願意接受外部依賴、要快收效 → D (denoiser)
           - 拒絕外部依賴、願意投資長期 → B (ReSTIR)
           - 兩條路都不滿足 → 接受現狀 / 重新評估場景設計
```

## 結論一句話

```
別跳實作、先診斷
G + H 兩三天、若命中、整段結束
若沒命中、再選 F → B/D 升級
ReSTIR 是大投資、不該作為起點
```
