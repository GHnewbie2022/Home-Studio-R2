## 第一部分：RALPLAN-DR Summary（置頂）

### Principles（核心原則，3-5 條）【unchanged from iter 1】

1. **場景語義凌駕採樣優化**：CONFIG 1/2 吸頂燈獨佔（N=1）、CONFIG 3 吸頂燈物理拆除（N=10，吸頂不入池）為使用者 2026-04-20 裁定之物理事實，光源池成員資格由此語義決定，不可為了「通用 dynamic N」把吸頂塞回 CONFIG 3 池以湊數。
2. **對稱性不可破**：`selectPdf`、`pdfNeeForLight` 的 PDF 分母、primary-hit 與 Next Event Estimation（下一事件估計）enable gate、Multiple Importance Sampling reverse-NEE PDF 的分母，四處必須同步改為 `1.0 / uActiveLightCount`（或透過 `selectPdfArg` 參數傳入）；違反即落入 memory `feedback_pathtracing_nee_pool_gate_symmetry.md` 的半截開關陷阱。
3. **rollback uniform 是二分定位工具，不是業務 gate**：`uR3DynamicPoolEnabled` 必須以 runtime-impossible guard 建構（對齊 memory `feedback_pathtracing_dce_sink_gate_trap.md`），避免 GLSL compiler Dead Code Elimination（死碼消除，以下稱 DCE）掉靜態 11-slot 路徑。
4. **防呆優先於精緻**：`uActiveLightCount = 0`（全部關閉）必須回傳黑色、不進入取樣分支；`int(rng() * N)` 的 clamp、LUT 未填滿區段的初值，皆須明確寫定，不依賴「不會發生」假設。
5. **不變約束（memory 硬紀律）**：對使用者嚴禁技術縮寫（必寫完整英文 + 白話中文）、每改遞增 cache-buster、每步 Playwright MCP 視覺驗證、bug 強制走 `/systematic-debugging`、Port 9001 URL 不加資料夾前綴、scene setupGUI 禁用 mouseControl 守門。

1. **CONFIG 1/2 底噪必須回到 R2-18 純吸頂燈基準**：使用者驗收 R3-6 時已明確點名「金屬反射銳利度判讀受阻」，這是本階段存在的唯一理由，其他優化（例如 power-proportional importance sampling）一律延後。
2. **CONFIG 3 吸頂燈雙重退場**：光源池踢除（物理能量不疊加）+ UI brightnessCtrl.disable() + 更名「吸頂主燈（CONFIG 3 已拆除）」，兩者缺一即語義破損；UI 層必須與光源池成員資格的 ceilingInPool gate 同一個 source of truth。
3. **Multiple Importance Sampling power heuristic β=2 不回歸**：動態 N 改變 `selectPdf` 即改變 `pdfNeeOmega`，連帶改變 `w_nee / w_brdf` 權重。R3-6 fix04 才剛解除「Cloud 雙計風險」，此次修改必須驗證 N=1 / N=10 / N=11（rollback）三種狀態下 reverse-NEE PDF 計算與 heuristic 計算全鏈一致。

### Viable Options（≥ 2，含 bounded pros/cons）【iter 2 修訂：新增 Option D 作廢軌跡】

#### Option A：LUT + uActiveLightCount 雙 uniform（主規格方向，推薦）

- **機制**：
  - JS 端維護 `uActiveLightCount` (int) + `uActiveLightIndex[11]` (Int32Array LUT，slot → 原 11-pool idx)
  - Shader helper `sampleStochasticLightDynamic`：`slot = int(rng() * uActiveLightCount)` → `origIdx = uActiveLightIndex[slot]` → 用 `origIdx` 分派到原 11-slot 五條分支（ceiling / Track / Wide / Cloud）
  - `selectPdf = 1.0 / float(uActiveLightCount)` 於 helper 內部設定
  - `pdfNeeForLight` 【iter 2 修訂】改**簽名加 `float selectPdfArg` 參數**（原 iter 1 方案「內部讀 uActiveLightCount uniform」作廢），呼叫點依 dispatch 情境傳對應值
- **Pros**：
  - 物理語義乾淨：關閉的光源直接不進 LUT，有效樣本率 = 1.0
  - CONFIG 3 排除吸頂燈只需 LUT build 時 `ceilingInPool = (currentConfig !== 3)` 一條 gate
  - 與現有 shader 5 條分支（idx 0 / 1-4 / 5-6 / 7-10）兼容，dispatch 表保留
  - 【iter 2 修訂】selectPdfArg 顯式參數讓 pdfNeeForLight 呼叫處的 PDF 分母一目瞭然，rollback（呼叫 sampleStochasticLight11 路徑）與 dynamic 路徑不會共用全域 uniform 造成交錯污染
- **Cons**：
  - LUT 重建時機需嚴格管控（applyPanelConfig 尾端 + 四個 checkbox onChange 尾端），重建不即時會讓 checkbox toggle 出現 1-frame 錯採樣
  - shader 端 `uActiveLightIndex[11]` 為 `int[]` uniform，three.js pathtracing renderer 此 repo 無 int[11] 先例，需於 Step 1 以 throw-first assertion 驗證 `{ value: new Int32Array(11), type: 'iv' }` 或等價綁定方式成功（【iter 2 修訂】Step 1 驗收錨點擴增）
  - pdfNeeForLight 呼叫點有 6 處（L271 / L300 / L324 / L368 / L1028 / L1702），需逐一傳入正確 `selectPdfArg`

#### Option B：保留 11-slot 陣列、「關閉 slot throughput=0 早退」+ Russian-roulette 補償（替代方案，作廢）

- **機制**：不改 helper 簽名，保留 11 格骰子；關閉的光源 throughput = 0 直接 return，但在 `selectPdf` 分母乘上「啟用比例」`(N/11)` 還原機率平衡
- **Pros**：shader 改動最小、無 LUT uniform
- **Cons**：
  - **數學錯誤**：有效樣本率仍是 N/11，丟棄率沒消除，只是 selectPdf 名目變化——底噪 √(11/N) 倍惡化根因完全未解，違反 Decision Driver 1
  - Multiple Importance Sampling reverse-NEE PDF 需同步乘 `(N/11)` 修正，對稱改動比 Option A 更多
  - 即使配 Russian-roulette 補償，variance 下降幅度不可能達到 R2-18 基準
- **結論**：**作廢**，根因未解（Driver 1 違反）。

#### Option C：Power-proportional Light Picking（power-CDF，R3-7 規格，作廢）

- **機制**：按 flux_i / Σflux 建 Cumulative Distribution Function（累積分佈函數）表，sample 時做 CDF inverse 查表
- **Pros**：variance 最低（高流明光源權重高）、符合 Many-Light Sampling 教科書做法
- **Cons**：
  - 本階段目標是「有效樣本率 = 1」而非「加權優化」；先做 CDF 再做 dynamic N 會把兩個變更疊加，pre-mortem 失敗情境暴增
  - R3-6 plan 明確把 power-proportional 列為「Part A 可選擴充」，R3-7 再處理
  - CONFIG 1/2 N=1 情境下 CDF 退化為單點 probability 1.0，與 Option A N=1 退化等價，但多一層 CDF 資料結構維護成本
- **結論**：**範疇外作廢**，延後 R3-7 處理。

#### Option D：共用單函式 `sampleStochasticLightN` + git revert rollback（iter 2 新增，作廢）

- **機制**：不做 `sampleStochasticLight11` / `sampleStochasticLightDynamic` 雙函式並存，只留一份 `sampleStochasticLightN(selectPdfArg, activeCount, LUT, ...)`，rollback 時用 git revert commit 回到 11-slot 固定版
- **Pros**：shader 維護成本 × 1，helper 邏輯集中
- **Cons**：
  - rollback 需 re-compile shader（git checkout + 清 cache-buster），無法 runtime 二分定位（**違 Principle 3「rollback uniform 是二分工具」**）
  - 單函式內 `if (dynamic) ... else ...` 受 GLSL compiler const-fold 影響（若 compiler 發現 `uR3DynamicPoolEnabled` 某分支永遠為 false → DCE 掉——memory `feedback_pathtracing_dce_sink_gate_trap.md`）
  - 失去「整支函式都是 legacy」的清晰度，debug 時無法快速區分 dynamic path bug 與原 11-slot 共用邏輯 bug
- **結論**：**作廢**，保留決策軌跡於 ADR Alternatives。

**選定 Option A**；Option B / C / D 具體作廢理由已寫明，符合 RALPLAN-DR「若只剩 1 可行選項須明示替代方案作廢」要求。

### Pre-mortem（deliberate mode 必要：4 情境 × failure mode / root cause / early warning / mitigation）【iter 2 修訂：情境 3 mitigation 改寫 + 情境 1 race 擴充 + 新增情境 4】

#### 情境 1：checkbox toggle 後單 frame 黑屏 / 白閃【iter 2 擴充 race 細節】

- **Failure mode**：使用者在 CONFIG 3 下取消 Track checkbox，畫面瞬閃全黑或過曝 1-2 幀才回穩
- **Root cause**：LUT 重建發生在 `onChange` 尾端，但 `uActiveLightCount` 已先被改為新值（例如 10 → 6），shader 該幀 `int(rng() * 6)` 取到 slot 5，但 LUT 重建還沒把 slot 5 寫入 → 讀到未定義 idx（可能是舊值指向關閉的 Cloud rod 3）
- **【iter 2】race 細節**：「先寫 LUT 後切 count」是 JS 層對 GPU 的呼叫順序，並**無跨 draw call 原子性保證**。Three.js uniform 寫入實際發生在 `material.onBeforeRender` → `gl.uniform1iv` / `gl.uniform1i`。為讓 LUT 全部寫完才切 count 對 GPU 有效：
  - 寫入時機必須對齊 `wakeRender()` 之 `gl.drawArrays` 前一刻（`rebuildActiveLightLUT` 內序列化 LUT → count，再由 renderer 下一幀 flush 到 GPU）
  - `needClearAccumulation = true` 與 LUT/count 同幀生效，累積緩衝區清零不帶舊 variance
  - count 初值 = 舊值、LUT 全部寫完後**單一 count assignment** 切新值（避免中間態被 renderer snapshot）
- **Early warning**：首次切換 checkbox 時 console 出現 `pickedIdx out of LUT range` 警告；Playwright 錄影抓到單幀亮度突變 > 30%
- **Mitigation**：
  - LUT 重建封裝為 `rebuildActiveLightLUT(source)` 單一函式，**先填完 LUT 全 11 格、再 single-assignment 寫 `uActiveLightCount`**
  - 函式尾端必呼叫 `needClearAccumulation = true` + `wakeRender()`
  - shader helper 首行加 `if (uActiveLightCount <= 0) { throughput = vec3(0); return nl; }` 防呆

#### 情境 2：CONFIG 1 → CONFIG 3 → CONFIG 1 切換後，吸頂燈 brightnessCtrl 卡在 disable 狀態【unchanged from iter 1】

- **Failure mode**：使用者從 CONFIG 3 回切 CONFIG 1，吸頂主燈 UI 仍灰色不可拖，且 `.name()` 還顯示「吸頂主燈（CONFIG 3 已拆除）」
- **Root cause**：`applyPanelConfig(3)` 尾端的 `.disable() + .name()` 有寫；但 `applyPanelConfig(1/2)` 沒對稱補 `.enable() + .name('吸頂主燈') + .setValue(900)` → UI state machine 缺 reverse 路徑（對齊 memory `feedback_pathtracing_nee_pool_gate_symmetry.md` 半截開關陷阱）
- **Early warning**：切換完立刻手動拖 brightness slider 無反應；`brightnessCtrl._disabled === true` 在 console 查得出來
- **Mitigation**：
  - applyPanelConfig 內以 `if (config === 3) { disable + name('... 已拆除') } else { enable + name('吸頂主燈') + setValue(900) }` 雙向補齊
  - 加 Playwright contract test：三路徑切換（1→3→1 / 2→3→2 / 3→1→3）每次均驗 brightnessCtrl.domElement 的 disabled attribute

#### 情境 3：Multiple Importance Sampling reverse-NEE PDF 與 dynamic N 失配 → Cloud 雙計火球重演【iter 2 修訂：mitigation 改 selectPdfArg 參數】

- **Failure mode**：CONFIG 3 Cam 2/3 長時間收斂後 Cloud rod 下方地板/西牆出現「比 R3-6 更亮」的光暈，或 N=1 CONFIG 1 吸頂燈下方出現非預期 firefly
- **Root cause**：修改了 `sampleStochasticLightDynamic` 的 `selectPdf = 1.0/N`，但 shader `pdfNeeForLight` 內部仍用 `1.0/11.0` 字面（L240），或呼叫點 L1028 / L1702 的 `pNeeReverse = pdfNeeForLight(...)` 傳錯分母 → Multiple Importance Sampling power heuristic `w_nee / w_brdf` 用錯分母，w_brdf 過大，BSDF-indirect 命中 emitter 時同光源貢獻再累加（Cloud 雙計風險重演）
- **Early warning**：
  - N=1 退化路徑驗收時，Cam 1 畫面與 R2-18 像素級比對出現 > 5% 亮度偏差
  - 開啟「R3-6 MIS 啟用」checkbox 前後亮度差異異常（正常應為 variance 減少、均值不變）
- **【iter 2 修訂】Mitigation**：
  - 改 `pdfNeeForLight` 簽名**加入 `float selectPdfArg` 參數**（iter 1 方案「內部讀 uActiveLightCount uniform」作廢，避免耦合 + 便於 rollback branch 獨立傳不同 selectPdf）
  - 新簽名：`float pdfNeeForLight(vec3 x, vec3 lightPoint, vec3 lightNormal, float lightArea, float selectPdfArg)`
  - 函式體內 `return (1.0 / 11.0) * (dist2 / (cosLight * safeArea));` 改 `return selectPdfArg * (dist2 / (cosLight * safeArea));`
  - 呼叫點傳參規則：
    - `sampleStochasticLight11` 內 L271 / L300 / L324 / L368 呼叫 → 傳 `1.0 / 11.0`（保留 legacy 路徑正確性）
    - 新 `sampleStochasticLightDynamic` 內對應 4 處呼叫 → 傳 `1.0 / float(uActiveLightCount)`
    - BSDF-indirect reverse Multiple Importance Sampling 的 L1028（lampCenter, ceiling）/ L1702（cloudTarget, Cloud）兩處 → 依當前 dispatch 傳對應值：`uR3DynamicPoolEnabled > 0.5 ? 1.0 / float(uActiveLightCount) : 1.0 / 11.0`
  - Grep 雙 pattern `1\.0\s*/\s*11\.0` 驗收：改動後實證命中數 = **1**（僅 L256 `sampleStochasticLight11` 內 `selectPdf = 1.0 / 11.0` 保留；L240 pdfNeeForLight 內字面已消）

#### 情境 4：shader 編譯失敗 / uniform1iv 於 WebGL2 ES 3.00 binding 失敗【iter 2 新增】

- **Failure mode**：首次 `browser_navigate` 載入 Home_Studio.html 後畫面全黑，或 console 出現 `INVALID_OPERATION: no bound value for uniform 'uActiveLightIndex'`
- **Root cause**：three.js 對 array uniform 預設型別判定為 float。若直接寫 `uActiveLightIndex: { value: [0,1,2,...,10] }`（普通 JS Array），three.js 會視為 `uniform1fv` 綁定，但 shader 宣告 `uniform int uActiveLightIndex[11]` → 型別不匹配，WebGL2 link error 或 sampler/int implementation-defined behavior 取到 0。必須顯式指定：
  ```javascript
  uActiveLightIndex: { value: new Int32Array(11).fill(-1), type: 'iv' }
  ```
  或等價 three.js r150+ 作法（視 renderer 版本）。本 repo 無 `int[11]` uniform 先例（僅 `uCloudFaceCount` 為純量 int），是首次。
- **Early warning**：
  - `browser_navigate` 完成後 console 出現 `getShaderInfoLog` 報錯或 `uniform1iv binding failed`
  - 首幀全黑但無 GLSL compile error → 已綁定但型別錯誤
- **Mitigation**：
  - Step 1 `initSceneData` 內 uniform 宣告後立即 throw-first assertion：
    ```javascript
    if (!(pathTracingUniforms.uActiveLightIndex.value instanceof Int32Array) ||
        pathTracingUniforms.uActiveLightIndex.value.length !== 11) {
      throw new Error('[R3-6.5] uActiveLightIndex uniform 型別 / 長度錯誤');
    }
    ```
  - `gl.uniform1iv` 綁定失敗時 console 立即報錯（非靜默）
  - Playwright `browser_evaluate` 驗 `pathTracingUniforms.uActiveLightIndex.value instanceof Int32Array && length === 11`

### ADR（Architecture Decision Record）【iter 2 修訂：Decisions 段 + Alternatives 補 Option D】

- **Decision**：採 Option A「LUT + uActiveLightCount 雙 uniform」+ `sampleStochasticLightDynamic` shader helper + ceilingInPool gate，搭配 rollback uniform `uR3DynamicPoolEnabled` 保留 R3-6 靜態 11-slot 路徑以利二分定位。
- **【iter 2 新增】Decisions（iter 1 Open Questions 已決）**：
  - **Q1 決議**：`pdfNeeForLight` 改**簽名加 `float selectPdfArg` 參數**（非「內部讀 uActiveLightCount uniform」）。理由：顯式參數讓 PDF 分母於呼叫處一目瞭然，rollback branch（呼叫 `sampleStochasticLight11` 路徑）與 dynamic branch 傳不同值，不會共用全域 uniform 造成交錯污染；便於 BSDF reverse MIS 兩處依當前 dispatch 傳對應值。
  - **Q3 決議**：observability 採 **GUI label 即時顯示 N 值**（Critic NICE TO HAVE 升級為採納）。Step 5 追加 `activeLightCountDisplay` read-only textCtrl，CONFIG / checkbox 切換後使用者可在 GUI 面板直接讀 N 值，無需開 DevTools。
- **Drivers**：
  1. CONFIG 1/2 底噪必須回到 R2-18 基準
  2. CONFIG 3 吸頂燈物理拆除須雙重退場（光源池 + UI）
  3. Multiple Importance Sampling power heuristic β=2 不回歸 Cloud 雙計
- **Alternatives considered**：
  - Option B「11-slot 保留 + throughput=0 早退 + Russian-roulette 補償」：作廢。有效樣本率仍是 N/11，根因未解，違反 Driver 1
  - Option C「Power-proportional CDF」：作廢。範疇外（屬 R3-7 規格）
  - **【iter 2 新增】Option D「共用單函式 sampleStochasticLightN + git revert rollback」：作廢**。rollback 需 re-compile shader、無法 runtime 二分定位（違 Principle 3「rollback uniform 是二分工具」）；單函式內業務分支受 GLSL DCE 風險；失去「整支函式都是 legacy」的清晰度
- **Why chosen**：Option A 物理語義乾淨（關閉的光源直接不在池內，有效樣本率 = 1.0），對稱改動範圍可預測（LUT 重建 5 個觸發點 + pdfNeeForLight 6 處呼叫點 + shader helper dispatch 表 + 10 處 NEE 呼叫點），rollback 路徑可以 runtime uniform 切換不必 re-compile shader。
- **Consequences**：
  - 正面：CONFIG 1/2 底噪回歸 R2-18 基準；CONFIG 3 吸頂燈雙重退場明確；selectPdfArg 顯式參數讓 rollback 與 dynamic path PDF 分母分離清晰
  - 負面：新增 3 個 uniform（`uActiveLightCount` + `uActiveLightIndex[11]` + `uR3DynamicPoolEnabled`）；shader helper 複製為 dynamic 版保留舊版，維護成本 × 2 直到 R3 全部驗收；pdfNeeForLight 6 處呼叫點均須改傳 5 實參
  - 連動：R3-7 indirectMul 校準、R3-8 採購評估 4 情境切換，會依賴 dynamic N 行為穩定
- **Follow-ups**：
  - FU1：GPU timestamp query 統計 per-light-class pickedIdx 分佈（observability O2 延後項，R3-7/R3-8 視需要）
  - FU2：R3-7 power-proportional CDF 上線後，`uActiveLightIndex` 改為存 CDF 權重（Option C 延伸合流）
  - FU3：R3 全部驗收後 merge `r3-light` → `main` 前，決定是否移除 `sampleStochasticLight11` legacy helper
