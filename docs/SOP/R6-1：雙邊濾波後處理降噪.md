```

### 重要教訓（為未來階段保留）

```
教訓 1：Cross-bilateral 5×5 不是 denoise，是 spatial blur with edge preservation
  - sigma_color 高斯衰減的本質是「顏色相近 weight 高」
  - 對「均勻雜點」只能 blur 不能去除（顏色仍在閾值內）
  - 對「螢火蟲」weight 趨近 0 反而保留 outlier
  - 想要真正 denoise 需要：時序累積 reproject、SVGF、OIDN AI
  - memory：feedback_pathtracing_bilateral_blur_not_denoise.md

## A. RALPLAN-DR 摘要

### A.1 Principles

```
P1（執行域唯一）：所有降噪運算限定 HDR-linear per-sample 域
                 （× uOneOverSampleCounter 之後、ScreenOutput 之前）
                 禁止在 Reinhard / gamma 之後再進濾波。
                 註：Bloom brightpass uniform 鎖定 pathTracingRT 不切，
                     bilateral 結果只進 ScreenOutput 不進 Bloom（C-1 決策案 a）。

P2（既有降噪不動）：保留 ScreenOutput_Fragment.glsl 行 1~345 的 walk denoiser 結構不改
                  （邊緣偵測、m37 鄰域、edgePixelColor blend、cameraIsMoving 分支全留）。

P3（邊緣訊號權威）：centerPixel.a / nextToAnEdgePixel 是現成且權威的邊緣訊號，
                 任何新增 bilateral 必尊重它，不得自行重算邊緣。

P4（sigma 必有推導）：所有 sigma_color preset 數值必須有 noise σ 物理推導，
                  禁止 hardcoded 魔數，禁止以「合理即可」收尾。

P5（GUI 失效零代價）：bilateral toggle 關閉時等同 main 分支零差異，
                  不可在 shader 內以 if-else 切；改用 RT pointer 切換。
```

### A.2 Decision Drivers（top 3）

```
D1（不動既有 walk denoiser 的回滾安全性）
   權重最高。R3 累積大量 fix01~fixN 已知該 denoiser 對邊界穩定的依賴。
   若 bilateral pass 出問題，必須能在 1 commit 內回到 main 分支視覺一致。

D2（HDR-linear 物理域正確性）
   bilateral 的 sigma_color 物理意義是 radiance noise σ，
   只能在 per-sample radiance 域操作；接到 Reinhard 後值會被壓縮、sigma 失去物理意義。

D3（C3/C4「朝外光」cosLight ≈ 0 的 variance 才是真痛點）
   §R5-Z 已證實 C3/C4 NEE 採樣 50%+ 浪費（cosLight ≈ 0）→ variance 高，
   平坦區肉眼可見的「螢火蟲」與「色斑」需要的是 noise σ 抑制，
   非 detail preservation；bilateral 對這個場景剛好對口。

   ※ ⚠ 此 D3 之 C3/C4 朝外光分類已過期（2026-05-01 校正）：
     實際 C3 朝外光成立、但 C4 不屬朝外光（軌道燈朝下投射、廣角燈漫射）。
     「朝外光 cosLight ≈ 0」之 variance 痛點僅完整適用 C3、C4 痛點來源不同。
     本段保留作 R6-1 撤回時點之歷史紀錄、若重啟 D3 論證須重新分類。
```

### A.3 Viable Options（公平比較全部 5 案）

| 案 | 簡述 | 優 | 劣 |
|---|------|----|----|
| **A** | 取代式：在既有 walk denoiser 的 m37 內換成 bilateral kernel | 改動行數最少（~30 行） | 違反 P2，回滾代價高，walk 邊緣處理被一併取代 |
| **B** | 雙寫：walk denoiser 結果 + bilateral 結果疊加 | 邊緣保護雙保險 | 雙倍 fragment cost；兩 kernel 互相干擾語意難分析 |
| **C** | 並排新增 PostDenoise pass（推薦） | walk denoiser 不動；GUI off 時換 RT 指向即等同 main；HDR-linear 域純粹 | +1 RT（1080p +33 MB / 4K +133 MB），多 1 次 fragment 採樣 |
| **D** | 最小侵入：cross-bilateral 接在 walk denoiser **後**、Bloom 前 | 改動最小，walk 結果當 guide texture | walk 已過 `*= uOneOverSampleCounter`、近邊緣已 blend；sigma_color 域不再純，物理推導斷裂 |
| **E** | À-Trous SVGF 簡化版：以 centerPixel.a / nextToAnEdgePixel 當 edge-stop function 多級 wavelet | 平坦區降噪量上限最高 | 多 pass、多 RT、實作量 ≥3× 方案 C；維護成本爆增 |

### A.4 推薦方案

**採方案 C（並排 PostDenoise pass）。**

理由：
1. P1 / P2 / P5 全滿足，是唯一同時滿足三條的方案。
2. D1 回滾路徑最乾淨——把 screenOutputUniforms.tPathTracedImageTexture 指回原 RT、刪 1 個 RT、移 1 段 shader 就完整回滾。
3. 工時可壓在 3.5 工作天內（D / E 都會超）。

### A.5 D / E 排除理由（公平 invalidation）

- **D 排除**：執行域不純。walk denoiser 行 355 已 `*= uOneOverSampleCounter` 並做了 edgePixelColor blend，該結果已不是純 radiance（近邊緣已 mix 過 centerPixel）；在這之後做 cross-bilateral 的 sigma_color 物理推導斷裂——sigma 對應的不再是 NEE pool variance，而是「混血像素」，我們無法估算 sigma 數值。違反 P4。
- **E 排除**：À-Trous SVGF 多級 wavelet 至少需 3~5 次 pass、3~5 個 RT，光是 RT 規格 + ping-pong 管理就超 3.5 工作天上限。後續若 C 證明降噪不夠，可再升級為 E（保留為 follow-up，但不在 R6-1）。

### 步驟 1a：JS reference bilateral kernel + contract-test【0.4 天】

**改動檔案**：
- 新增 `docs/tests/r6-1-bilateral-kernel.test.js`（~120 行）

**輸入**：
- 純 JS function `bilateralKernel(centerRGB, neighbors, sigmaColor, sigmaSpatial)`
- 輸入：中央像素 RGB、鄰域 RGB+offset 陣列（5×5 = 25 樣本，跳過中央剩 24）
- 輸入：sigmaColor、sigmaSpatial 兩標量

**輸出**：
- 純 JS function 回傳 weighted average RGB

**驗收標準（可機器驗證）**：
1. 全黑鄰域 + 全黑中央 → 回 (0,0,0) abs err < 1e-9
2. 鄰域全等於中央 → 回中央色 abs err < 1e-9
3. 中央 (1,0,0)、鄰域全 (0,0,0)、sigmaColor=0.001 → 回 ≈ (1,0,0) rel err < 1e-3（color stop 阻止 blend）
4. 中央 (0.5,0.5,0.5)、鄰域全 (0.5,0.5,0.5)+ε noise (σ=0.05)、sigmaColor=0.1、sigmaSpatial=2.0 → 回值 σ < 0.02（noise 至少壓 60%）
   （附註：本 case 為 lab condition，scene σ 估值取 0.05；步驟 4 (I) 量化指標不沿用此門檻，以實測 200×200 平坦區 noise σ 為準）
5. spatial weight 對稱性：(+1,+1) 與 (−1,−1) 同色鄰域 weight 完全相同

**風險守門**：
- 若 driver case 4 noise σ 壓不到 0.02，**停**——回看 sigmaColor 推導表（見 §F）
- 純 JS 階段就要 PASS，不能拖到 GLSL 階段才發現公式錯

### 步驟 1b：JS sigma_normalize function + contract-test【0.2 天】

**改動檔案**：
- 同檔追加（~40 行）

**輸入**：純 JS function `sigmaNormalize(sigmaBase, sampleCount)`

**輸出**：`sigmaBase × Math.max(1.0, 4.0 / Math.sqrt(sampleCount))`

**驗收標準**：
1. N=1 → 回 4 × sigmaBase（max 取 4，因 4/√1=4）
2. N=16 → 回 1 × sigmaBase（max 取 1，因 4/√16=1）
3. N=64 → 回 1 × sigmaBase（max 取 1，4/√64=0.5 < 1）
4. N=10000 → 回 1 × sigmaBase（floor）
5. monotonic decreasing in N for N ≤ 16

**風險守門**：
- max(1, …) floor 必須在；否則 N 大時 sigma 趨 0 → bilateral 退化成只取中央 → 等於關閉降噪

**GLSL bilateral pass 規格**：
```glsl
// PostDenoise_Fragment.glsl 偽碼
void main() {
    vec2 fc = gl_FragCoord.xy;
    vec4 centerPixel = texelFetch(tPathTracedImageTexture, ivec2(fc), 0);
    vec3 centerRGB = centerPixel.rgb * uOneOverSampleCounter;  // P1：HDR-linear per-sample 域

    // 邊緣 pixel 直通（Architect 修訂 #4 + Critic v2 B1 累積域對稱修訂）
    // 寫回域與下方 sumW>0 路徑保持對稱（先 × 再 ÷），統一語意供 SVGF 升級接手
    if (centerPixel.a == 1.0) {
        pc_fragColor = vec4(centerRGB / max(uOneOverSampleCounter, 1e-8), centerPixel.a);
        return;
    }

    // sigma normalize（Architect 修訂 #3）
    float sigmaC = uSigmaColor * max(1.0, 4.0 / sqrt(max(uSampleCounter, 1.0)));
    float sigmaS = uSigmaSpatial;

    vec3 sumRGB = vec3(0.0);
    float sumW = 0.0;

    // 5×5 鄰域
    for (int dy = -2; dy <= 2; dy++) {
        for (int dx = -2; dx <= 2; dx++) {
            vec4 nPixel = texelFetch(tPathTracedImageTexture, ivec2(fc) + ivec2(dx, dy), 0);
            vec3 nRGB = nPixel.rgb * uOneOverSampleCounter;

            // 鄰域邊緣 spatial weight × 0（Architect 修訂 #4）
            float edgeMask = (nPixel.a == 1.0) ? 0.0 : 1.0;

            float dC = length(nRGB - centerRGB);
            float dS = length(vec2(dx, dy));
            float wC = exp(-(dC * dC) / (2.0 * sigmaC * sigmaC + 1e-8));
            float wS = exp(-(dS * dS) / (2.0 * sigmaS * sigmaS + 1e-8));
            float w = wC * wS * edgeMask;

            sumRGB += nRGB * w;
            sumW += w;
        }
    }

    vec3 outRGB = (sumW > 1e-6) ? (sumRGB / sumW) : centerRGB;

    //!! WARNING: 寫回域為 N × per-frame-denoised，不是純 path tracer 累積值
    //!! 下游 ScreenOutput * uOneOverSampleCounter 才能還原顯示空間
    //!! R6-2/R6-3 SVGF 接此 RT 必須先 / uSampleCounter 還原 per-sample 域
    //!! Bloom brightpass uniform 鎖定 pathTracingRT 不切（C-1 案 a），不從本 RT 讀
    pc_fragColor = vec4(outRGB / max(uOneOverSampleCounter, 1e-8), centerPixel.a);
}
```

> 寫回值乘回 `1 / uOneOverSampleCounter` 是為了讓 ScreenOutput 行 355 的 `*= uOneOverSampleCounter` 維持原邏輯不變（GUI off 時 RT 指向切換等同 main 分支零差異 = P5）。

**驗收標準（可機器驗證）**：

   const mean = pixels.reduce((s, p) => s + p, 0) / pixels.length;

7. 結果寫入 SOP 表格：
   ```
   | Cam | Config | mean | std | σ_measured | σ_predicted (§F.6) | deviation | sigma base 是否校正 |
   ```

## D. ADR 雛形（v3 收尾終版）

### Decision

採方案 C：在 ScreenOutput pass 之前並排新增 PostDenoise pass（獨立 RenderTarget + 獨立 fragment shader），執行域鎖定 HDR-linear per-sample 域（× uOneOverSampleCounter 之後、ScreenOutput 之前）。bilateral 公式：

```
sigma_color_eff = sigma_color_preset × max(1, 4 / sqrt(uSampleCounter))
邊緣 pixel（centerPixel.a == 1.0）採對稱 / N 寫法直通（先 × 再 ÷，統一兩條路徑語意）；
鄰域邊緣 pixel spatial weight × 0；
GUI toggle 經 RT 指向切換，不在 shader 內 if-else；
Bloom brightpass 鎖定 pathTracingRT 不切（C-1 案 a）。
```

### Drivers

D1（不動既有 walk denoiser 的回滾安全性）/ D2（HDR-linear 物理域正確性）/ D3（C3/C4 cosLight ≈ 0 的 variance 才是真痛點）

### Alternatives considered

**主支線：**
| 案 | 排除理由 |
|---|---------|
| A 取代式 | 違反 P2，回滾代價高，walk 邊緣處理被取代 |
| B 雙寫 | fragment cost 翻倍，兩 kernel 語意干擾 |
| D 最小侵入（walk 後 cross-bilateral） | 執行域不純，walk 結果已 mix，sigma_color 物理推導斷裂，違反 P4 |
| E À-Trous SVGF 簡化版 | 多 pass / 多 RT，工時遠超 3.5 天上限；保留為 R6-2+ 升級路徑 |

**設計支線（C-1）：**
| 分支 | 決策 |
|------|------|
| 案 a：Bloom 故意不享 denoise（採用） | bloomBrightpassUniforms 鎖 pathTracingRT，省同步點維護債 |
| 案 b：Bloom 也享 denoise（延後） | +0.1 天 + 三處同步點，升級為 follow-up F4 |

**邊緣寫回支線（C-3）：**
| 分支 | 決策 |
|------|------|
| 對稱 / N 寫法（採用） | 與 sumW>0 路徑統一語意，便於 SVGF 升級接手 |
| 直接寫 centerPixel.rgb | 省 1 次乘除但語意不對稱，reviewer 需重新驗等價性 |

### Why chosen

C 是唯一同時滿足 P1 / P2 / P5 的方案，回滾路徑乾淨（改 1 個 RT 指向 + 刪 1 段 shader），工時 3.4 天 < 3.5 天上限。

### Consequences

正面：

## F. sigma_color 物理推導

> 使用者非工程背景，本節以「白話 + 必要算式」並陳。

### F.1 per-sample noise σ 從 NEE pool 推算

**前提（從 r3-5b-cloud-area-nee.test.js + §R5-Z）**：
- NEE pool size = 11（7 R3-5a lights + 4 Cloud rod faces）
- selectPdf = 1/11 ≈ 0.0909
- C1/C2 朝下光（吸頂燈）：cosLight ≈ 1，NEE 採樣有效率 ≈ 100%
- C3/C4 朝外光（軌道燈、Cloud rod）：cosLight 多半 ≈ 0，NEE 採樣有效率 ≈ 50%（§R5-Z 估值）

> ⚠ **上述 C3 / C4 朝外光分類已過期**（2026-05-01 校正）：
> 實際 C3 = 只開 Cloud 漫射燈（朝外光成立、Cloud rod 4-rod 朝側面噴）
>      C4 = 只開軌道燈 + 廣角燈、Cloud 關閉（軌道燈朝下投射、廣角燈漫射、皆不屬朝外光）
> 「朝外光」分類只能適用 C3、不適用 C4。本前提保留作 R6-1 撤回時點之歷史紀錄、不改原文。
> 若日後重啟 §F.1 變異推導、須以正確 C3 / C4 定義重算 cosLight 與 NEE 採樣有效率。

**白話解釋**：
- 每根 ray 進場景，要選一盞燈做 shadow ray 試算（NEE，也就是向光源直接發射投影射線）
- 朝下光：選到後 shadow ray 朝上射，幾乎一定打到光，每次都「有貢獻」→ pixel 值穩定
- 朝外光：選到後 shadow ray 朝側射，但光是橫的、能量主要往側面噴；如果你這 pixel 的點剛好「不在光的橫面正前方」，cosLight 算出來幾乎 0，貢獻 ≈ 0 → 該 sample 等於白幹一次
- 「白幹」的比例越高，per-sample 之間的差距（變異）越大 → 視覺上就是 noise

### F.2 變異係數 CV 量級

對 fully NEE-driven path（採 R3-5b 註解 + r3-5b test [J] 假設）：

```
C1/C2 cosLight ≈ 1：每 sample contribution 變化小，CV ≈ 0.3
C3/C4 cosLight 約 50% 接近 0：half sample = 0、half sample 正常 → CV ≈ 1.0
```

> CV = std / mean。CV ≈ 1 是「標準差等於平均值」，從統計上對應「螢火蟲」與「亮塊」級別的可見噪點。

### F.3 per-sample noise σ 對應 sigma_color base

**核心關係**：
- 假設場景平均 radiance L ≈ 0.5（HDR-linear，post-tonemap 落 0.3 灰）
- per-sample noise σ_C12 = CV_C12 × L ≈ 0.3 × 0.5 = 0.15
- per-sample noise σ_C34 = CV_C34 × L ≈ 1.0 × 0.5 = 0.5
- 累積到 sample N 後 noise σ_N = σ_per_sample / sqrt(N)

**bilateral sigma_color 設計準則**：
- sigma_color 應該對應「同一表面真實顏色變化的下限」，要剛好夠大才能 blur 過 noise σ_N，但不能大到吃掉真實 albedo 變化
- 經驗 ratio：sigma_color_eff ≈ 1.5 × σ_N（夠 blur noise 但留 detail margin）
- 給定 N=16（典型穩態），sigma_color_eff ≈ 1.5 × 0.15 / 4 = 0.056 → C1/C2 base 取 0.05 ✅
- 給定 N=16，sigma_color_eff ≈ 1.5 × 0.5 / 4 = 0.19 → C3/C4 base 取 0.20 ✅

### F.4 sample_count normalize 公式

```
sigma_color_eff = sigma_color_base × max(1, 4 / sqrt(N))
```

物理意義：
- N = 1：noise σ_N = σ_per_sample（最髒），sigma 放大 4 倍
- N = 16：noise σ_N = σ_per_sample / 4，sigma 設為 base（剛好對齊上面 F.3 推導）
- N ≥ 16：noise 已收斂到 base 等級，sigma 不再縮（floor at base）

> 為何 floor at base 不繼續縮？因為 sigma 縮到 0 等於 bilateral 退化成只取中央像素 = 等於關閉降噪；而我們希望即使 N 很大，bilateral 仍能消除 firefly 殘留（outlier 仍存在）。

### F.5 C1/C2 vs C3/C4 sigma 差異從 cosLight 量綱解釋

```
sigma_C12 / sigma_C34 = CV_C12 / CV_C34 = 0.3 / 1.0 = 0.3
→ sigma_C12 base = 0.3 × sigma_C34 base = 0.3 × 0.20 = 0.06 ≈ 0.05 ✅（C1）
→ C2 加軌道輔光，CV 略升 → 0.08
→ C4 加軌道輔光，CV 略升 → 0.22
```

物理本質一句話：**朝外光的「白幹比例」高於朝下光，sigma_color 必須等比放大才能蓋過 variance**。

|--------|---------------|-----|-----------------------|------------------|
| C1 | ≈ 1 | 0.3 | 0.15 | **0.00** ⚠ |
| C2 | ≈ 0.7（混合） | 0.5 | 0.25 | **0.00** ⚠ |
| C3 | ≈ 0~0.3 | 1.0 | 0.50 | **0.20** |
