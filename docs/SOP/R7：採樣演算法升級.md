> 狀態：⏳ 規劃大綱、未啟動

> 觸發條件：R6-3 視覺收斂優化全段執行完仍達不到「C3/C4 30~60 秒收乾淨」目標時啟動

```
R6 三主線執行完之後若仍未達標、表示：
```

## 2026-05-06 / 2026-05-07 優先序與命名修正

```
R6-3 結案後，先不要直接進 WebGPU 整套搬遷。

原因：
  1. three.js WebGPU 移植會牽動材質與 shader 路線。
  2. 官方 migration 文件指出 ShaderMaterial / RawShaderMaterial 這類自訂材質不支援。
  3. 本專案 path tracer 目前高度依賴自訂 shader。
  4. 因此 WebGPU 整套搬遷不是換 renderer 名稱，會變成整條管線重寫。

短期順序：
  1. R7 甲 blue noise sampling 升級。
     小、可逆、直接針對低 SPP 觀感。

  2. R7 乙光源 importance sampling 機率優化。
     範圍仍在既有 R3-6 / R3-6.5 架構內，但會碰 MIS，需要嚴格合約測試。

  3. R7 丙 WebGPU / hybrid preview 小型概念驗證。
     只驗證「移動時乾淨 preview，停止後回正式收斂」。
     不做整套搬遷。

暫緩：
  R7 丁 ReSTIR 與 R7 戊 path guiding 重做。
  兩者都屬論文級工程，先保留為後段選項。

參考：
  https://threejs.org/manual/en/webgpurenderer
```

## 候選方向（依 2026-05-07 R7 新命名）

### 方向甲：blue noise sampling 升級（第一順位）

```
做什麼（白話）
  現在 RNG（隨機數產生器）已用 IQ Shadertoy hash + Jacco Bikker
  但 blue noise 的時序分佈仍有調整空間
  例如改用 Heitz & Belcour 2019 的「Spatiotemporal Blue Noise」

為什麼先做
  低 spp 階段（0~64 spp）肉眼觀感、blue noise 分佈品質直接影響「看起來舒不舒服」。
  它改動最小、可逆、驗收最快。
  就算無法真正乾淨，也能快速知道低 SPP 顆粒能不能變得比較不刺眼。

工時估    3~5 天
風險      低、屬可逆替換
驗收      C3 / C4 1 / 2 / 4 / 8 SPP 對照；看黑白點是否比較不刺眼，靜止收斂不可走樣
```

### 方向乙：光源 importance sampling 機率優化（第二順位）

```
做什麼（白話）
  既有 R3-6.5「Many-Light Sampling 動態 NEE pool」已是 stochastic light selection
  R7 重新審視 pool 的機率分佈、是否能依「光源到像素的距離 / 立體角」做更好的權重
  屬「在既有架構內改進」、不像 ReSTIR 是換架構

為什麼排第二
  它比 blue noise 風險高，但仍比 ReSTIR / path guiding 小很多。
  若 blue noise 改善有限，下一步才值得碰光源機率。

工時估    1~2 週
風險      動到 R3-6 既有 MIS、需與既有平衡點對齊
驗收      C1~C4 光源狀態不可走樣；C3 / C4 低 SPP 與 1024 SPP 都要 A/B
```

### 方向丙：WebGPU / hybrid preview 小型概念驗證（第三順位）

```
做什麼（白話）
  做小頁面或小分支，專門驗證移動時的乾淨 preview。
  停止移動後才回正式 path tracing 收斂。

為什麼不直接整套搬遷
  three.js WebGPU 官方 migration 文件提醒，ShaderMaterial / RawShaderMaterial 不支援。
  本專案現有 path tracer 幾乎都靠自訂 shader。
  整套搬遷會變成 render pipeline 與 shader 架構重寫。

工時估    1~2 週做出能判斷方向的小型概念驗證；完整搬遷不估
風險      小型驗證可控；整套搬遷風險高
驗收      移動時乾淨、順、不閃廉價灰模；停止後可回正式品質收斂
```

### 方向丁：ReSTIR（多光源時空複用採樣，暫緩）

```
做什麼（白話）
  現在每個像素每 frame 都重新挑一盞光源算亮度貢獻
  挑光源這步運氣不好就會出現雜點
  ReSTIR 改成「每像素留一個小水塘、記住目前為止挑過的最好光源樣本」
  跟周圍像素 + 上一 frame 的水塘合併
  等於「自動學會哪盞光對這個像素重要」、不再亂挑

為什麼適合本專案
  C3 = 只開 Cloud 漫射燈（4 條燈條、4-rod 多光源）
  C4 = 只開軌道燈 + 廣角燈（Cloud 關閉）
  C3 與 C4 採樣特性不同（漫射 vs 方向性直接光）、ReSTIR 對多光源場景的挑樣變異是業界新標準

工時估    3~5 週（NVIDIA 2020 之後新標準、論文級實作）
風險      會動到 R3-6 既有多重重要性採樣（MIS）邏輯、改動範圍大
參考      Bitterli 2020 ReSTIR 原論文 + 後續 ReSTIR GI / Restir PT 衍生
```

### 方向戊：path guiding 重做（Müller 2017 級，暫緩）

```
做什麼（白話）
  現在反彈光線的方向、是依「物件表面的反射特性（材質）」決定
  path guiding 多加一層學習：「這個位置的光線、過往哪些方向比較容易接到光源」
  把這份學習記錄起來、下次抽方向時偏向那些方向、雜點降很多

為什麼適合本專案
  暗角 / 遠離光源區域、間接光彈跳變異大
  path guiding 對「室內 + 多次反彈」場景效益顯著

工時估    4~6 週（含學習結構設計 + GPU 友善資料結構）
風險      跟 R6-3 階段 3 的 path guiding 部分重疊、需確認介接層
參考      Müller 2017 Practical Path Guiding + 後續 NN-based 衍生
```

## 範圍邊界（守門禁忌）

```
1. R7 不啟動之前、嚴禁先動 R3-6 既有 NEE pool / MIS heuristic
   理由：避免 R6-3 階段執行中誤踩 R7 範疇、整段失控

2. R7 啟動必先 ralplan 共識
   ReSTIR / path guiding 屬論文級實作、不可繞過共識直接 executor

3. R7 任一方向上線、必須對 R3-6 / R3-6.5 既有體例做 regression 驗證
   既有「Cloud rod / 軌道燈 / 4 燈條」C1~C4 視覺結果不可走樣

4. R7 屬「大階段」、合併到 main 必依 R 主線 DONE 紀律
   R7-? 子階段完工屬本地 commit、整段 R7 完工才合併 main + push

5. 嚴禁把 R7 當 R6-3 的 fallback 投錢
   若 R6-3 已能達 30~60 秒目標、R7 不啟動、屬下世代專案範疇
```

## R6-3 → R7 銜接決策樹

```
R6-3 結案完成
   ↓
路徑 1 先做最小可逆改善
   → R7 甲 blue noise sampling 升級
   → 若低 SPP 顆粒變得比較不刺眼，保留並評估是否足夠
   → 若改善有限，進路徑 2

路徑 2 在既有採樣架構內優化
   → R7 乙光源 importance sampling 機率優化
   → 必須保護 R3-6 / R3-6.5 MIS 合約
   → 若仍無法改善移動低 SPP 視覺障礙，進路徑 3

路徑 3 小型 hybrid preview 概念驗證
   → R7 丙 WebGPU / hybrid preview 只做小頁面或小分支
   → 驗證移動時乾淨 preview，停止後回正式收斂
   → 不做整套搬遷

路徑 4 論文級採樣重構
   → R7 丁 ReSTIR
   → R7 戊 path guiding 重做
   → 只有路徑 1 / 2 / 3 都無法達標，且使用者接受大工程時才啟動

明確暫緩
   → WebGPU 整套搬遷
   → R7 丁 ReSTIR
   → R7 戊 path guiding 重做

明確決策節點
   每個方向啟動前，先定義可回退條件與 C1~C4 regression。
   每次只做一個方向，避免多個變因混在一起。
```

## 不在範圍

```
1. WebGPU 整體架構升級
   理由：three.js WebGPU 目前對 ShaderMaterial / RawShaderMaterial 移植有限制；本專案自訂 shader 比例高。
   允許：只做小型 hybrid preview 概念驗證。

2. 自寫 AI 降噪模型訓練（成本不對、不如等 OIDN 進步）
3. 多 GPU 分散運算（單機 demo 用不到）
4. 離線 baking pre-computed light（屬不同類別演算法、改變專案定位）

5. 飛利明Varilumi 燈具採購     使用者自理、不在 AI 介入範圍
```
